# BizAudit 核心系统设计方案

## 一、 统一语言 (Ubiquitous Language)
在整个系统的生命周期中（包括需求讨论、架构图绘制、提示词编写），严格使用以下核心领域词汇：

* **Tech Defect (技术缺陷):** 底层静态扫描工具发现的客观代码问题（如异常未捕获、精度丢失）。
* **Business Context (业务上下文):** 提取自 BDD `.feature` 文件或 DDD 领域文档的纯文本业务红线与规则。
* **Impact Evaluation (影响评估):** 大模型结合技术缺陷与业务上下文后，产出的最终业务影响诊断结论。
* **Severity Level (严重等级):** 评估结论的标准化量级（如：致命、高、中、低、无影响）。
* **Audit Task (审查任务):** 代表一对“缺陷-上下文”的完整处理生命周期。

---

## 二、 战略设计：限界上下文划分 (Bounded Contexts)
为了防止系统变成一团乱麻，我们将系统划分为四个边界清晰的上下文。核心域以外的上下文均作为“防腐层（ACL）”存在。

1.  **缺陷摄取上下文 (Defect Ingestion):**
    * **职责:** 负责与外部的 Linter、SonarQube 等工具打交道。无论外部工具输出的是 JSON、XML 还是 SARIF 格式，该上下文都负责将其“清洗”并翻译为系统内部标准的 `Tech Defect` 模型。
2.  **业务发现上下文 (Context Discovery):**
    * **职责:** 负责在代码仓库的物理目录中，根据缺陷所在的文件路径，向上或同级“寻址”并提取对应的 BDD/DDD 业务文档（`Business Context`）。
3.  **核心审查上下文 (Core Audit) —— 【核心域】:**
    * **职责:** 系统的“大脑”。负责将缺陷与业务规则对齐，制定审查策略，并得出业务爆炸半径的评估结果。
4.  **交付上下文 (Delivery):**
    * **职责:** 决定结果以何种形态展示给用户（例如：渲染为命令行的彩色文本、生成 PR 评论的 Markdown、或是封装为 MCP 协议的 JSON 响应）。

---

## 三、 战术设计：核心域建模 (Core Domain Modeling)
聚焦于“核心审查上下文”，我们定义以下业务模型（不涉及任何数据库表或 JSON 结构，纯粹是业务对象）：

### 1. 聚合根 (Aggregate Root)
* **模型:** `审查任务 (Audit Task)`
* **特征:** 具有唯一生命周期标识（Task ID）。它包裹着一个缺陷和对应的业务规则。
* **行为:**
    * 开启评估流程。
    * 记录评估结果并扭转自身状态（待处理 -> 评估中 -> 已完成/失败）。

### 2. 实体 (Entities)
* **模型:** `技术缺陷 (Tech Defect)`
* **特征:** 具有逻辑唯一性（通常由“文件路径 + 行号 + 错误签名”计算得出 Hash 值作为 ID）。即使报错内容相同，发生在不同文件的缺陷也是不同的实体。

### 3. 值对象 (Value Objects)
* **特征:** 完全由其包含的属性定义，没有唯一 ID，不可变。
* **模型列表:**
    * `代码定位 (Code Location):` 包含文件物理路径与行号。
    * `业务规则 (Business Rule):` 包含所属的领域名称（如“支付域”）以及纯文本的规则描述。
    * `评估结论 (Evaluation Result):` 包含 `严重等级 (Severity Level)` 枚举值以及大模型生成的 `业务影响说明 (Impact Description)` 文本。

### 4. 领域服务 (Domain Service)
* **模型:** `影响评估器 (Impact Assessor)`
* **职责:** 当评估动作过于复杂，不适合塞进单一实体时使用。它负责将缺陷与业务规则组装成特定的策略（Prompt 骨架），并指挥大模型能力进行推理。

## 四、 架构分层设计：六边形架构 (Hexagonal Architecture)
系统采用依赖倒置的洋葱/六边形架构，确保**核心业务逻辑绝对不依赖任何具体的技术栈或外部大模型 API**。

### 第 1 层：领域层 (Domain Layer) - 最内层
* **内容:** 上文“战术设计”中定义的所有聚合根、实体和值对象。
* **原则:** 绝对纯净，没有任何外部依赖，完全是业务抽象。

### 第 2 层：应用层 (Application Layer) - 用例编排
* **内容:** 审查工作流协调器 (Audit Workflow Orchestrator)。
* **职责:** 定义输入/输出的**接口标准（Ports）**。编排标准流程：获取缺陷 -> 找文档 -> 触发评估 -> 输出结果。
* **原则:** 只发号施令，不亲自干活。它通过接口调用基础设施层。

### 第 3 层：基础设施层 (Infrastructure Layer) - 适配器 (Adapters)
* **内容:** 实现应用层定义的接口，负责真正的“脏活累活”。
    * `Linter 报告解析器:` 读取本地硬盘上的 ESLint JSON 文件。
    * `本地文档扫描器:` 用正则或文件系统 API 抓取 `.feature` 文本。
    * `LLM 网关适配器:` 封装对外部大模型（如 Gemini、Claude、OpenAI）的网络请求与 Token 管理。

### 第 4 层：接入层 (Presentation Layer) - 触发入口
* **内容:** 触发应用层工作流的机制。
    * *形态 A:* `CLI 命令行解析器`（接收开发者的终端命令）。
    * *形态 B:* `MCP Server 监听器`（接收 Cursor 等 AI IDE 的协议调用）。

---

## 五、 标准工作流时序 (Workflow Sequence)

当一次审查被触发时，系统的数据流转过程如下：

1.  **触发阶段:** 接入层（如 CLI）接收指令，唤醒应用层的协调器。
2.  **数据摄取:** 协调器调用基础设施，扫描当前项目的代码审查产物，生成一组 `技术缺陷` 实体列表。
3.  **上下文补齐:** 针对每一个缺陷，协调器调用本地扫描器，在代码相邻目录寻址，提取出对应的 `业务规则` 值对象。如果找不到，则该缺陷降级处理或跳过。
4.  **组装与推理:** 协调器将缺陷与业务规则装载入 `审查任务` 聚合根。领域服务介入，调用 LLM 网关，执行“技术与业务的交叉推理”。
5.  **结果沉淀:** LLM 返回结果被解析为 `评估结论` 值对象，附属在聚合根上，任务标记为已完成。
6.  **交付输出:** 协调器将所有完成的任务结果打包，交还给接入层，由其决定是打印在屏幕上还是发往前端。
