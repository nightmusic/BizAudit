# BizAudit 使用手册

## 1. 文档目的

本文档面向 BizAudit CLI 的使用者，说明如何准备输入、执行命令、理解输出结果，以及当前版本的注意事项。

BizAudit 是一个面向业务影响分析的命令行工具。它会读取静态扫描报告，尝试在代码附近查找业务文档，再结合缺陷信息给出业务影响评估结果。

## 2. 运行前提

### 2.1 环境要求

- Node.js 18 及以上
- npm
- 项目依赖已安装

如当前仓库尚未安装依赖，请先执行：

```bash
npm install
```

### 2.2 当前版本的启动方式

当前仓库未配置可执行脚本或全局命令，建议直接通过 `ts-node` 运行：

```bash
npx ts-node src/presentation/cli.ts --help
```

## 3. 核心能力

BizAudit 当前支持以下能力：

- 读取本地 Linter JSON 报告
- 将报告中的问题转换为内部缺陷对象
- 从缺陷文件所在目录向上查找业务文档
- 基于缺陷信息和业务文档生成影响评估
- 以文本或 JSON 形式输出结果
- 按严重等级进行过滤

## 4. 输入数据要求

### 4.1 报告文件

`audit` 命令要求传入一个本地 JSON 报告文件路径：

```bash
npx ts-node src/presentation/cli.ts audit ./reports/eslint-report.json
```

### 4.2 支持的报告结构

当前实现按 ESLint 风格 JSON 结构解析，格式示例如下：

```json
[
  {
    "filePath": "src/modules/order/service.ts",
    "messages": [
      {
        "line": 18,
        "ruleId": "no-floating-promises",
        "message": "Promises must be awaited."
      },
      {
        "line": 42,
        "ruleId": "eqeqeq",
        "message": "Expected '===' and instead saw '=='."
      }
    ]
  }
]
```

注意事项：

- 顶层必须是数组
- 每个文件对象需要包含 `filePath`
- `messages` 必须是数组
- 只有同时包含 `ruleId` 和 `line` 的消息才会被纳入审查

## 5. 业务文档发现规则

BizAudit 会从缺陷文件所在目录开始，逐级向上查找业务文档，直到当前工作目录为止。

当前会识别以下文件：

- `*.feature`
- `domain.md`
- `README.md`

找到第一份符合条件的文档后，系统会直接使用其全文作为业务上下文，不会继续向上查找。

## 6. 命令说明

### 6.1 查看帮助

```bash
npx ts-node src/presentation/cli.ts --help
```

### 6.2 执行审查

```bash
npx ts-node src/presentation/cli.ts audit <reportPath>
```

示例：

```bash
npx ts-node src/presentation/cli.ts audit ./reports/eslint-report.json
```

## 7. 命令参数

### 7.1 `--format`

指定输出格式，支持：

- `text`：终端彩色文本输出
- `json`：JSON 输出

示例：

```bash
npx ts-node src/presentation/cli.ts audit ./reports/eslint-report.json --format json
```

### 7.2 `--severity-filter`

按严重等级过滤输出，只保留“大于等于指定等级”的已完成结果。

可选值：

- `none`
- `low`
- `medium`
- `high`
- `fatal`

示例：

```bash
npx ts-node src/presentation/cli.ts audit ./reports/eslint-report.json --severity-filter medium
```

说明：

- 已完成任务会按等级过滤
- 失败任务不会被过滤掉，仍会显示
- 未完成或无结果的任务也会保留，便于排查问题

### 7.3 `--api-key`

用于传入大模型 API Key：

```bash
npx ts-node src/presentation/cli.ts audit ./reports/eslint-report.json --api-key your_key
```

也可以通过环境变量传入：

```bash
set LLM_API_KEY=your_key
npx ts-node src/presentation/cli.ts audit ./reports/eslint-report.json
```

### 7.4 `--config`

用于传入自定义配置文件路径：

```bash
npx ts-node src/presentation/cli.ts audit ./reports/eslint-report.json --config ./bizaudit.config.json
```

注意：当前版本中该参数仅预留，尚未实际生效。

## 8. 输出说明

### 8.1 文本输出

默认输出为终端文本报告，包含：

- 缺陷规则标识
- 文件位置
- 原始错误消息
- 识别到的业务域
- 影响评估说明
- 汇总统计

严重等级包括：

- `FATAL`
- `HIGH`
- `MEDIUM`
- `LOW`
- `NONE`

### 8.2 JSON 输出

当使用 `--format json` 时，输出结构类似：

```json
[
  {
    "id": "defect-id",
    "status": "COMPLETED",
    "defect": {
      "file": "C:\\project\\src\\modules\\order\\service.ts",
      "line": 18,
      "rule": "no-floating-promises",
      "message": "Promises must be awaited."
    },
    "severity": "MEDIUM",
    "impact": "该缺陷可能导致业务流程异常中断，建议人工复核。",
    "errorReason": null
  }
]
```

字段说明：

- `id`：缺陷任务 ID
- `status`：任务状态
- `defect`：缺陷原始信息
- `severity`：评估出的严重等级
- `impact`：影响评估描述
- `errorReason`：失败原因，仅任务失败时有值

## 9. 任务状态说明

当前任务状态主要包括：

- `PENDING`：待处理
- `IN_PROGRESS`：评估中
- `COMPLETED`：评估完成
- `FAILED`：评估失败

在当前实现里，如果缺陷已进入审查流程，通常最终会表现为：

- 成功时为 `COMPLETED`
- 失败时为 `FAILED`

## 10. 典型使用流程

### 场景一：本地查看彩色报告

```bash
npx ts-node src/presentation/cli.ts audit ./reports/eslint-report.json
```

适合在终端中快速查看审查结果。

### 场景二：只关注中高风险问题

```bash
npx ts-node src/presentation/cli.ts audit ./reports/eslint-report.json --severity-filter high
```

适合在问题较多时优先聚焦高风险项。

### 场景三：输出 JSON 供其他系统消费

```bash
npx ts-node src/presentation/cli.ts audit ./reports/eslint-report.json --format json
```

适合集成到脚本、CI 或其他工具链。

## 11. 常见问题

### 11.1 提示 `Report file not found`

原因：传入的报告文件路径不存在。

处理方式：

- 检查路径是否正确
- 建议传入相对当前目录可访问的文件
- 必要时改用绝对路径

### 11.2 提示 `Failed to parse linter report`

原因：报告文件不是合法 JSON，或结构不符合当前实现要求。

处理方式：

- 检查 JSON 语法是否正确
- 确认顶层是否为数组
- 确认每个结果项是否包含 `messages`

### 11.3 没有识别到业务文档

原因：缺陷文件路径向上的目录中未找到 `.feature`、`domain.md` 或 `README.md`。

处理方式：

- 在相关模块目录下补充业务文档
- 确认运行命令时的当前工作目录是否正确

说明：未找到业务文档时，系统会降级为基于技术信息的评估。

## 12. 当前版本限制

使用前请注意以下事实：

- 当前 `LlmGatewayAdapter` 仍为模拟实现，并未真正调用外部大模型服务
- 因为 LLM 返回的是模拟文本，严重等级解析通常会回退为 `MEDIUM`
- `--api-key` 参数当前已接收，但不会真正发起远程模型调用
- `--config` 参数暂未启用
- 当前以串行方式处理任务，未做并发优化
- 当前业务文档发现策略较简单，只取首次命中的文件全文

因此，当前版本更适合作为 CLI 原型、流程联调工具或后续接入真实模型前的基础骨架。

## 13. 建议目录组织

为了让 BizAudit 更容易识别业务上下文，建议将代码与业务文档按模块邻近放置，例如：

```text
project/
  src/
    order/
      README.md
      service.ts
      validator.ts
  reports/
    eslint-report.json
```

或者：

```text
project/
  src/
    payment/
      payment.feature
      handler.ts
```
